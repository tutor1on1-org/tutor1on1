# LESSONS

Durable cross-session lessons and operating rules moved from AGENTS.md (memory-maintenance refactor).

- Tutor branching rule: do not ask students to type navigation commands like "another question" or "summarize". Drive tutor branching through explicit `Learn` / `Review` / `Summary` buttons; session auto-start and the `Learn` button must dispatch `learn`, the `Review` button must dispatch `review`, and prompt output must never auto-route the student between learn/review/summary modes.
- Summary stability rule: cache/reuse summary when no new graded review result exists after last summary, and clamp weak-evidence re-summarize downgrades (for example `a<=1`) to avoid large mastery swings.
- New-device recovery rule: provide an explicit one-way "Take Server Copy" path for students that clears local session/progress data, resets session/progress download cursors + list ETags/run-state, and runs forced server->client pull without upload.
- Empty-local bootstrap rule: if local student `chat_sessions`/`progress_entries` are empty but secure-storage sync cursor/etag state exists, clear session/progress download cursor + domain sync state before delta pull; stale cursors can force `since` pagination past history and yield a false "no server data" empty import.
- Session/progress conflict rule: apply last-modified arbitration on both download and upload paths (compare local `syncUpdatedAt`/`updatedAt` vs remote `updated_at`) so the newer record wins and stale-side updates are skipped.
- Course sync correctness depends on server-authoritative versioning and deletion events: treat remote bundle version as source of truth, and replay deletion events on login to clean multi-device local state.
- Workflow-critical UI feedback should use persistent, manually dismissible messages instead of auto-fade snackbars, including marketplace enrollment/quit/download flows.
- Marketplace course identity must be normalized and enforced server-side as `(teacher_id + course_name_key)` to prevent duplicate course rows across repeated uploads.
- Marketplace visibility must require at least one bundle version; deleting the last bundle version should auto-unpublish the course to avoid stale/duplicate listings.
- JWT secret exposure is critical: with HS256 auth, anyone holding the secret can mint bearer tokens for arbitrary `sub`; rotate secrets immediately if exposed and minimize distribution.
- JWT rotation rollout must keep signing and verification separated: sign with current `JWT_SECRET`, verify with `[JWT_SECRET + JWT_PREVIOUS_SECRETS]` during cutover, and enforce `RECOVERY_TOKEN_ECHO=false` when `APP_ENV=production`.
- Stale local `remoteCourseId` links after server deletions can trigger bundle ensure failures; server must return `404 course not found` (not 500), and client upload flow must re-resolve/create remote course before `ensureBundle`.
- SSH key path for remote ops in this environment is `C:\\Users\\kl\\.ssh\\id_rsa`; use `-i` and `-o IdentitiesOnly=yes` explicitly.
- Canonical remote login tuple for this project: `ecs-user@43.99.59.107` with key `C:\\Users\\kl\\.ssh\\id_rsa` (key-only auth; do not do password/no-key retries).
- Bundle ensure recovery: when stale `course_id` is sent, server `EnsureBundle` should accept `course_name` fallback to resolve/create course and catalog row, preventing teacher upload failures after server-side deletions.
- After `EnsureBundle` fallback, client must use the returned `course_id` (not stale local ID) for subsequent calls like publish/update visibility, or upload flow can still fail with `course not found`.
- Student login sync must tolerate older servers missing `/api/enrollments/quit-requests` (treat 404 as empty) and auto-remove legacy assigned courses whose local teacher record is missing (equivalent to auto-approved quit cleanup).
- Student marketplace download/enrollment import must resolve server `teacher_id` to a local teacher user and persist that on `course_versions.teacher_id`; using student ownership can trigger legacy cleanup on relogin and hide still-enrolled courses.
- Archive stream lifecycle rule: when decoding ZIPs via `InputFileStream` + `ZipDecoder`, do not close the stream until all lazy entry reads are complete (validation/extraction/metadata read); close in `finally` after these steps.
- Student bundle re-download must reuse the existing local course mapped by `remoteCourseId` and apply `override` mode with server subject as course name; importing from extracted folder basename creates duplicate local courses and splits progress.
- Progress sync should be uploaded in batch (`/api/progress/sync/upload-batch`) instead of one-by-one requests to avoid per-minute sync rate-limit bursts.
- Teacher quit/deletion event replay must delete only the affected student assignment/data; never delete the teacher's local course definition based on "no assignments", or teacher course catalogs can silently shrink.
- Deletion-event replay regression tests should assert cursor behavior (`sinceId` forwarding + cursor advance) and role-specific cleanup outcomes (student login removes local assigned course; teacher login keeps teacher course and removes only affected student assignment/progress).
- When session sync import cannot resolve `remoteCourseId`, resolve local course by normalized subject (strip `_<timestamp>` suffix) before creating a new placeholder to prevent duplicate local course rows.
- Teacher upload preflight must run in this order: local course validation, semantic hash compare with latest remote bundle, then KP diff (added/deleted/updated) confirmation before upload when hash differs.
- Async cleanup rule: in `try/finally`, do not `return` an un-awaited `Future` that still depends on resources deleted in `finally` (use `await` first), or temp files can be removed before downstream read.
- Bundle creation must include only required course assets (`contents/context`, referenced lecture `.txt`, and prompt `.txt` files plus prompt metadata payload), excluding unrelated files to keep semantic hash stable.
- Local plugin path overrides must stay signature-compatible with upstream platform interfaces (for example, `record_platform_interface` permission API changes), or release builds can fail unexpectedly.
- Progress sync E2EE must upload/store ciphertext envelopes (`envelope` + `envelope_hash`) and decrypt client-side; keep compatibility by accepting legacy plaintext rows when envelope is absent.
- User key lifecycle must re-encrypt and upsert an existing local key pair with the current login password during `syncNow`; otherwise password changes can leave server key blobs undecryptable on new devices.
- Go toolchain is installed at `C:\\Program Files\\Go\\bin\\go.exe` (`go1.26.0`); if `go` is not recognized in PowerShell, invoke the absolute path or prepend `C:\\Program Files\\Go\\bin` to `PATH` for that session.
- Prompt composition precedence must remain `system override -> course append -> student append`; keep scope-resolution checks scripted so ordering regressions are caught early.
- Bundle extraction/import regressions should assert the returned extraction path is immediately import-ready (required files present and zero KP diff versus source bundle) to guard missing-await failures.
- Integration tests for auth-gated UI flows can stay deterministic/offline by using `AuthGate` with in-memory `AppDatabase`, a fake `AuthController`, and mocked `path_provider` channels for app/secure-storage paths.
- API authorization regressions should cover `/api/enrollment-requests` and `/api/bundles/download` with JWT identities plus DB checks; reject teacher/self-unenrolled access with `403` and assert successful download responses include `X-Accel-Redirect`.
- `EnsureBundle` stale-link behavior should be regression-tested at API level: stale `course_id` without `course_name` returns `404`, while providing `course_name` must resolve/create and return the effective `course_id` in response.
- Client upload flow must resolve remote target via `ensureBundle` and use the returned `course_id` for subsequent visibility updates; keep this enforced through `TeacherMarketplaceUploadService` tests.

- For server sync operations triggered from auth/home screens, use a blocking gray overlay with a bottom progress panel so users see sync phase and cannot trigger conflicting actions during sync.

- App-change handoff discipline: run self-battle, concrete build, validation, memory update, commit, push, and publish/upload the remote release in the same turn unless the user explicitly skips a step; do not report app work as done while any of those default steps are still pending or blocked without calling that out; see `WORKFLOW.md` for the detailed sequence.

- Public client snapshot rule: when publishing source only to prove client behavior, export a client-only snapshot from tracked files, exclude `remote/` and release/deploy helpers, never ship any hardcoded offline admin/bootstrap credential in the public client tree, and keep the official auth base URL overrideable via `--dart-define` instead of hardcoding it as the only possible target.

- Git staging discipline: default to `git add -A` before commit so user-visible work is not accidentally omitted; if a file is truly temporary or log-only and should remain untracked, add it to `.gitignore` instead of depending on selective staging.

- Student tutor chat reliability should enforce structured-output schemas for `learn` / `review` / `summary`, single-flight dedupe by `(session + prompt + call_hash)`, explicit retry telemetry (`attempt/reason/backoff`), and summary cache reuse when no new graded review evidence exists.

Logout and current-device deletion must pass through the same gate as app quit so students cannot bypass enforcement by leaving the signed-in screen without the teacher PIN.
- Teacher control PIN secrecy rule: student devices must never cache or compare `control_pin_hash` locally. When teacher-enforced study mode needs a PIN check, send the entered PIN to the server for verification against the current effective study-mode decision.
- macOS study-mode enforcement rule: use the soft study-mode path plus a blur/watchdog recovery timer that re-shows the app, refocuses it, and reapplies fullscreen/always-on-top after the delay. Do not try to fake a Windows-style workstation lock on macOS.
- Global close-button rule: every page-level screen, including startup/loading/error scaffolds, should expose the shared close action. Inject it through one shared AppBar helper so the button stays universal and always means global app quit, never a page-specific session-close variant.
- LLM JSONL metadata decryption rule: decrypt with the exact `(owner_user_id, owner_role, plaintext password)` tuple from the active login context; read `owner_user_id` / `owner_role` from raw JSONL first, then run `tool/decrypt_llm_logs.dart`. JSONL only stores encrypted metadata, so rendered prompts/full responses still require the local `llm_calls` DB when available.
- Simplified tutor prompt rule: bundled `learn` / `review` / `summary` prompts must ignore legacy `prompt_templates` DB overrides/appends. Old teacher-managed prompt rows can silently reintroduce incompatible JSON contracts during live runs.
- Simplified summary rule: current-session tutor summary must be derived from session evidence, not reused `progress_entries` from prior sessions. Cache only the current session's own summary, and compute `lit` locally from the session review counters.
- Live LLM retry rule: treat transient TLS `HandshakeException` the same as socket/timeout/http transport failures in `_sendWithRetry`, or otherwise-valid live validation can fail before the app even reaches prompt/schema handling.
- Structured review visibility rule: when `question_open=true`, the full student-visible question must appear in `teacher_message`, not only in `question_text`. In live runs the model can otherwise return `teacher_message="Try this."` with the real question hidden in JSON, and the student sees an unusable prompt.
- Prompt-improvement review rule: judge prompt quality from the student's visible transcript first, not from stylistic preference. For each bad turn, identify the exact user-facing failure, trace whether it comes from prompt wording vs app contract, and propose the smallest prompt change that directly blocks that failure mode.
- Simplified tutor contract rule: the active student flow should use only bundled `learn` and `review` prompts. `learn` returns `{text,difficulty,mistakes,next_action}`, `review` returns the same fields plus `finished`, there is no active `summary` prompt, and the app owns `summaryLit` locally from finished review counts.
- Live tutor validation rule: for Albert-style live regression runs, validate the simplified flow directly from `chat_sessions.summary_lit` plus `evidence_state_json` on a copied DB. Do not call a synthetic summarize action once summary is removed from the runtime contract.
- Weighted KP-pass rule: app-controlled `lit` should be derived from persistent KP passed counts, not a hard-coded session review count. Store teacher-managed weights per `(courseVersionId, studentId)` with defaults `easy=0.25`, `medium=0.5`, `hard=1`, `threshold=1`, and compute both session and progress `lit/lit_percent` from that weighted score after each finished review.
- KP-pass dialog rule: the tutor "passed" celebration must be driven by the same global `progress_entries` row that decides KP pass status. Persist a one-shot control-state event only when the current review turn changes the global KP from not-lit to lit, and populate the dialog counts from the post-update global `{easy,medium,hard}_passed_count` values, not from session-local rebuilt evidence.
- Review-hint prompt rule: for compound review items, a hint should advance only one unresolved sub-step and end with one short direct ask. Do not restate the whole original multi-part question after the hint unless the student would otherwise lose the task.
- Structured streaming delta rule: when streamed JSON fragments are rejoined for the final buffer, emit the exact same normalized delta to the UI/database callback instead of the raw provider fragment. The JSON reassembler must preserve standalone space fragments and may only dedupe duplicated boundary whitespace; it must not invent missing spaces inside split words.
- Tutor-visible sanitization rule: raw LLM payloads may contain inline reasoning markers such as `<think>...</think>`. Keep the raw payload for logs/debugging, but strip those blocks from the student-visible tutor text in both streaming-prefix extraction and final display resolution.
- LLM call-history rule: `llm_calls` must be append-only across retries. Do not overwrite earlier failed attempts by `call_hash`, or the LLM logs page will hide the exact bad payload that triggered the retry. Replay lookup may still resolve the latest row for a hash.
- Structured prompt hardening rule: when a model keeps mangling a small JSON contract, shorten the prompt, list the exact keys once, and include one valid JSON example that uses the exact key spelling. This reduces malformed-key retries better than adding more prose around the same schema.
- Visible-stream completion rule: the student can see streamed text from a failed first attempt before the app knows that attempt violated the structured contract. In that case `_sending` should stay active until the retry completes; diagnose apparent "finished text but still busy" reports by checking for an `APP retry` gap after the first streamed attempt.
- JSON streaming reassembly rule: never insert recovery spaces inside streamed JSON strings, whether they are object keys or visible value text. Token boundaries can split inside words, and providers can emit real spaces as standalone chunks, especially around number transitions, so JSON reassembly must preserve raw fragments and only collapse duplicated explicit boundary whitespace.
- Remote SSH access rule: for this server, use `ecs-user@43.99.59.107` with `C:\Users\kl\.ssh\id_rsa` and `-o IdentitiesOnly=yes`. If `root@...` or a local username fails with `Permission denied`, check the remote user before blaming the key. - Sidecar log identity rule: when local users are merged, mutable DB foreign keys can be rewritten but immutable JSONL sidecar logs keep the old local ids. Any log viewer that combines DB rows with JSONL rows must normalize teacher/student identity from current DB state, preferably by `call_hash`, instead of trusting raw sidecar ids.
- Device identity resilience rule: do not assume `Platform.localHostname` is safe on every release target. Reads used for default device naming or sync/device fingerprints must degrade to a stable fallback (for example OS name) instead of throwing before login/sync requests start, and auth flows must surface pre-request device/storage failures as visible errors instead of letting async button handlers fail silently.
- Backend release completeness rule: if any shipped change touches `remote/` behavior or schema contracts, the release is not complete until production DB migration is applied, the API binary is redeployed, `family-teacher-api.service` is restarted, and one live route check proves the new server contract is active. Do not assume client publish alone is enough just because public auth smoke still passes on the old server.

If the host source tree drifts behind the local repo and an on-host rebuild fails with missing symbols that are present locally, do not patch the host tree piecemeal under pressure; cross-compile the Linux binary from the validated local repo, upload it, and keep the exact backup dump/binary path in `WORKLOG.md`.

- OpenRouter compatibility rule: treat OpenRouter as its own OpenAI-compatible provider variant, not as raw OpenAI. Send reasoning via a top-level `reasoning` object, pass provider-specific discovery headers (`HTTP-Referer`, `X-OpenRouter-Title`) through shared header builders, and parse reasoning/usage from `reasoning_details` or `reasoning` plus `usage.completion_tokens_details.reasoning_tokens`.
