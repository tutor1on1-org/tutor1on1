# Program Architecture & Flow (for Codex)

This system is a deterministic orchestration layer (the program) plus a teacher brain (the LLM).
The program does **not** decide correctness, mastery, or transitions; it only executes the model’s
structured outputs, persists state, and supplies inputs.

## Core Concepts

### Modes
- LEARN: concept explanation and understanding estimation.
- REVIEW: practice question selection + interactive solving; produces mastery decisions.
- SUMMARY: concise final decision for this knowledge point.

### Turn State (universal)
- UNFINISHED: the current segment/question continues; next call should use a *_CONT prompt.
- FINISHED: the current segment/question ends; next call should use a *_INIT prompt (or SUMMARY).

### Prompts
- LEARN_INIT / LEARN_CONT
- REVIEW_INIT / REVIEW_CONT
- SUMMARY

## Data Model (minimal)

### Per knowledge point (kp_id)
- types: list of {type_id, ...} including OTHER (generated by teacher-side tooling; program enforces any count limits)
- error_book: aggregated by (type_id, mistake_tag)
  - wrong_count
  - last_seen_ts
  - (optional) example_refs / notes

### Per session (rolling)
- prev_json: last model JSON output (persisted)
- evidence (rolled by the model inside REVIEW JSON):
  - a: attempts_in_session
  - c: correct_in_session
  - h: last_hint_level
  - t: last_type_id
  - mt: recent mistake_tags (max ~3)

### Practice history summary (program-produced)
A compressed summary of recent practice outcomes to steer selection and explanation.
Keep it small and stable; it can include counts by type_id, recent outcomes, and typical mistakes.

## Control Flow (program)

### 1) Select which prompt to call
Inputs: current_mode, prev_json.turn_state, and whether you are starting a new segment/question.

- If prev_json.turn_state == UNFINISHED:
  - call MODE_CONT (LEARN_CONT or REVIEW_CONT)
- Else:
  - call MODE_INIT (LEARN_INIT or REVIEW_INIT), or SUMMARY if you are finalizing

### 2) Assemble model inputs
- INIT calls include full context: lesson_content, types, error_book_summary, practice_history_summary,
  and for REVIEW_INIT also presented_questions (candidate pool for this knowledge point).
- CONT calls are minimal:
  - recent_dialogue (must include the entire dialogue since the current INIT began; allow truncation by max_tokens)
  - prev_json (the previous turn output)

### 3) Call the LLM and parse JSON
- Parse strictly as JSON (no extra framing).
- teacher_message may contain Markdown and math; treat it as an opaque string for UI rendering.

### 4) Persist and apply updates
- Persist prev_json for next turn.
- If REVIEW and grading is present:
  - store per-question attempt record (question text, student answer, is_correct, hint_level, etc.)
- If error_book_update is present:
  - update error_book aggregation at (type_id, mistake_tag) by delta_wrong
  - store mistake_note as a short human-facing descriptor (can keep latest or a small set)

### 5) Execute transitions
- LEARN: follow next_mode (LEARN/REVIEW).
- REVIEW: follow next_mode (LEARN/REVIEW/SUMMARY).
- SUMMARY: follow next_step (RELEARN/CONTINUE_REVIEW/MOVE_ON) at the application level.

## Review Question Selection (where selection happens)
- The program supplies a pool: presented_questions (all candidates for the kp_id).
- The model (REVIEW_INIT) selects ONE best question and may lightly modify it to reduce repetition.
- The selected question is emitted in JSON.question.text and also included in teacher_message.

## UNFINISHED Handling
- For REVIEW_CONT:
  - If not a final answer: action is UNFINISHED, grading must be null, error_book_update must be null,
    and mastery_level must not change.
  - Only when FINISHED and grading exists can mastery_level change and error_book_update be applied.
- For LEARN_CONT:
  - Continue the same conceptual segment; do not start a new topic unless the segment is finished.

## Practical Notes
- Truncation: keep recent_dialogue as a sliding window; always include the INIT turn and the latest k turns.
- Safety: validate that model output contains only allowed keys; reject and retry (same prompt) if violated.
- Determinism: log (prompt version, inputs hash, output JSON) to reproduce sessions.
