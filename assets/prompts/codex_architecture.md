PROJECT CONTEXT SUMMARY
=======================

The goal of is to design and refine the interaction protocol between:
- an education App (state machine + persistence, no semantic understanding)
- and an LLM acting as a one-on-one tutor

The system is prompt-driven.
All pedagogical judgment, correctness checks, mastery estimation, and summarization
are made by the LLM via structured JSON outputs.
The App only parses, stores, and routes based on those outputs.

This discussion has converged on the final, in-use versions of 5 prompts:
- LEARN_INIT
- LEARN_CONT
- REVIEW_INIT
- REVIEW_CONT
- SUMMARY

These prompts and their JSON schemas are already integrated into the app and must NOT be changed.


APP–LLM INTERACTION LOGIC (HIGH LEVEL)
-------------------------------------

Core principle:
- The App does NOT understand meaning.
- The LLM understands meaning and makes all decisions.

The App:
- Stores `recent_dialogue`
- Stores the last LLM JSON output as `prev_json`
- Parses only specific JSON fields
- Chooses which prompt to call next based on those fields

The LLM:
- Decides whether the interaction is LEARN / REVIEW / SUMMARY
- Decides whether a segment/question is finished
- Judges correctness and mistakes
- Estimates understanding and mastery
- Generates practice summaries (because this requires semantic judgment)

Key routing logic used by the App:
- If `turn_state == UNFINISHED` → call the corresponding *_CONT prompt
- If `turn_state == FINISHED` → call *_INIT or SUMMARY depending on `next_mode`


PROMPT OVERVIEW
-------------------------------

1) LEARN_INIT
-------------
Purpose:
- Start a new learning segment for a single knowledge point.
- Focus on conceptual understanding, not formal presentation.

LLM responsibilities:
- Explain the core idea and intuition.
- Adapt explanation using error patterns and practice history.
- Judge conceptual understanding.
- Not too formal. 

Inputs provided by the App:
- kp_title         (program-filled)
- lesson_content   (program-filled)
- understanding, error_book_summary & progress_summary (generated by LLM earlier)

JSON fields parsed by the App:
- understanding
- next_mode
- turn_state

Other fields:
- teacher_message is displayed only; never parsed.


2) LEARN_CONT
-------------
Purpose:
- Continue an unfinished learning segment.
- React only to the latest student message.

LLM responsibilities:
- Decide whether understanding has improved.
- Avoid prolonging the segment if understanding is already clear.

Inputs provided by the App:
- recent_dialogue  (program-filled)
- prev_json        (exact previous LLM output)

JSON fields parsed by the App:
- understanding
- next_mode
- turn_state

Important constraint:
- READY should not be downgraded unless a real misconception appears.


3) REVIEW_INIT
--------------
Purpose:
- Start a new practice question.

LLM responsibilities:
- Select ONE best question from the provided pool.
- Avoid repetition.
- Optionally make light modifications (numbers/wording).
- Assign a stable `type_id`.

Inputs provided by the App:
- presented_questions   (program-filled; full candidate pool)
- types                 (program-filled, includes OTHER)
- error_book_summary    (program-filled)
- practice_history_summary (program-filled, LLM-generated previously)

JSON fields parsed by the App:
- question
- turn_state
- next_mode
- evidence
- mastery_level

Notes:
- Question selection is semantic → LLM responsibility.
- App does not choose or modify questions.


4) REVIEW_CONT
--------------
Purpose:
- Continue the SAME practice question until it is finished.

LLM responsibilities:
- Decide whether the student has given a final answer.
- Grade correctness.
- Decide whether a mistake is trivial or meaningful.
- Update mastery and practice summary.

Key rules enforced by the prompt:
- If NOT final answer:
  - turn_state = UNFINISHED
  - grading = null
  - error_book_update = null
  - mastery_level MUST NOT change
- If final answer:
  - turn_state = FINISHED
  - grading filled
  - error_book_update only for non-trivial mistakes
  - mastery_level may update
  - next_mode chosen by LLM

practice_history_summary (IMPORTANT):
- Generated by the LLM here when a question finishes.
- Included in the same JSON output.
- The App stores it verbatim and feeds it back later.
- The App does NOT try to infer what counts as “practice”.

JSON fields parsed by the App:
- turn_state
- grading
- error_book_update
- evidence
- mastery_level
- next_mode
- practice_history_summary


5) SUMMARY
----------
Purpose:
- Produce a final decision for the current knowledge point.

LLM responsibilities:
- Make a clear, decisive judgment.
- Recommend what the app should do next.

Inputs provided by the App:
- understanding_history
- practice_history_summary
- error_book_summary
- last_evidence

JSON fields parsed by the App:
- mastery_level
- next_step

Constraints:
- No teaching.
- No new questions.
- No reopening unfinished items.


OVERALL DESIGN PHILOSOPHY
-------------------------
- Learn answers: “Does the student understand the concept?”
- Review answers: “Can the student apply it?”
- Summary answers: “Are we done with this knowledge point?”

- Correct answer + clear understanding ⇒ stop.
- No enforced formality.
- No semantic logic in the App.
- LLM reasons; App executes.
